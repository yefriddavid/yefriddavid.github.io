### Code Review - Hallazgos y Desviaciones de Estándares

He realizado un code review de los cambios actuales y he identificado varios puntos que no cumplen con los estándares definidos en `CLAUDE.md` o que podrían mejorarse para mantener la consistencia del proyecto:

#### 1. Ausencia de Notificaciones en Sagas (Incumplimiento Crítico)
Según `CLAUDE.md` (regla: *Toast notifications must be dispatched from sagas, not from views*), todas las operaciones asíncronas deben despachar un toast de éxito o error.
*   **Problema:** Los nuevos sagas en `src/sagas/system/` (`appSettingsSagas.js`, `auditLogSagas.js`, `fcmTokenSagas.js`, etc.) no están despachando acciones `notifications/push`.
*   **Impacto:** El usuario no recibe feedback visual cuando una operación (como guardar una configuración) termina correctamente o falla.

#### 2. Implementación Redundante de Redux (Inconsistencia)
El proyecto cuenta con un `crudFactory` para estandarizar acciones y reducers.
*   **Problema:** `appSettingsActions.js` y `appSettingsReducer.js` han sido implementados manualmente repitiendo la lógica de `crudFactory`. En cambio, `auditLogActions.js` sí lo utiliza.
*   **Recomendación:** Refactorizar `appSettings` para usar `createCRUDActions` y `createCRUDReducer`, añadiendo las excepciones necesarias mediante `extraCases`.

#### 3. Constantes en Vistas (Incumplimiento de Regla)
`CLAUDE.md` prohíbe definir constantes de dominio dentro de las vistas (regla: *Static domain constants must live in src/constants/*).
*   **Problema:** `src/views/taxis/Summary.js` todavía define `const MONTHS = [...]` de forma local.
*   **Recomendación:** Mover a `src/constants/commons.js` o usar el hook `useLocaleData` que ya se está usando en `TaxisHome.js`.

#### 4. Nomenclatura de Acciones
*   **Observación:** Se está utilizando el sufijo `successRequestFetch` y `errorRequestFetch`. Aunque `crudFactory` lo permite, es una nomenclatura poco común (normalmente es `fetchSuccess`). 
*   **Sugerencia:** Si es posible, estandarizar a un formato más corto y directo como `fetchSuccess` / `updateError`.

#### 5. Falta de Pruebas Unitarias
Se han eliminado correctamente los tests de `paymentVaucher` (ya que la funcionalidad fue removida), pero no se han agregado tests para los nuevos slices de Redux.
*   **Problema:** Slices como `appSettings` y `usageMetrics` no tienen cobertura de tests en `src/reducers/__tests__/`.

#### 6. Lógica de "Saving" en `AppVariablesSettings.js`
*   **Observación:** La vista gestiona un estado local `saving` para rastrear qué llave se está guardando. 
*   **Mejora:** El reducer de `appSettings` ya tiene un `lastSavedKey` y `saveSeq`. Se podría simplificar la vista confiando más en el estado global, aunque la implementación actual no es incorrecta, es algo compleja.

---

### Aspectos Positivos Identificados

*   **Migración a Redux:** Se ha cumplido correctamente con la regla de **"No direct API or database calls from views"** en casi todos los archivos refactorizados (`Visits.js`, `PushSubscribers.js`, `StorageSettings.js`, etc.).
*   **Uso de Componentes Estándar:** Se está usando correctamente `StandardGrid`, `Spinner` y `firestoreCall` siguiendo los wrappers del proyecto.
*   **Estructura de Carpetas:** Los nuevos archivos en `actions/system`, `reducers/system` y `sagas/system` siguen la arquitectura modular esperada.
